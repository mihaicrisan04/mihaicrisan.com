A RAG-powered document Q&A platform built as a university research project at Babes-Bolyai. The goal: let someone drop a stack of PDFs into the system and have a grounded conversation about their content, with citations back to the source chunks.

## Why

Most off-the-shelf RAG demos look good on plain prose and fall apart on real-world documents — PDFs with tables, slides with bullets that depend on the heading, scanned pages with OCR noise. Nara is the testbed for handling those properly: a proper parsing pipeline, structure-aware chunking, hybrid retrieval, and a UI that always points back at the source chunk so a researcher can verify the answer instead of taking the model's word for it.

## The pipeline

Uploads land in Convex storage, then a Python service picks them up and runs them through [Docling](https://github.com/docling-project/docling) — a document parsing library that converts PDF/DOCX/PPTX/images into clean Markdown while preserving structure (headings, tables, captions). The same service then chunks the markdown using Docling's `HybridChunker`, which respects document structure rather than blindly splitting on token count.

Each chunk lands in `documentChunks` with its metadata (headings, captions, page number). Convex indexes both vector embeddings and a full-text search field, so the retrieval layer can do hybrid search — vector similarity plus full-text RRF — without an external search engine.

## The stack

- **Frontend** (`apps/web`): Next.js + Tailwind + shadcn/ui, talking to Convex via generated types so the client/server contract can't drift
- **Backend** (`packages/backend`): Convex schema, functions, and the chat agent
- **Data ingestor** (`packages/data-ingestor`): FastAPI service in Python, runs Docling and pushes chunks back into Convex
- **Auth**: Better-Auth, integrated through Convex
- **Build**: Turborepo monorepo with Bun as the package manager

## Why this shape

Convex handles the reactive data layer well, but document parsing is a Python-native problem — Docling has no good JS port. Splitting the ingestion into a separate FastAPI service kept the parsing pipeline in its natural ecosystem while letting the chat experience stay fully reactive on the Convex side.
