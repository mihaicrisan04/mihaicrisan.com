This is the site you're on right now. It started as a standard portfolio template and turned into a long-running design playground.

## Why

Most portfolio sites are read-only — a CV, some links, maybe a blog. I wanted mine to be queryable: ask a real question about my work, get a grounded answer with the right project surfaced, citations and all. That one decision pulled the rest of the architecture along with it — RAG, a content pipeline keyed off MDX, embeddings rebuilt on every deploy. The minimalism on the surface is the part that took longest to get right.

## The redesign

The current iteration leans into minimalism — a single centered paragraph on the homepage, four inline interactive keywords, no shader background, no marquee. Each keyword has its own micro-interaction: a Dia-style iridescent shimmer on hover for one, a hand-drawn strikethrough that reveals an X logo for another, a popover with my live GitHub contribution chart for a third.

## The AI assistant

The portfolio has a built-in chat (Zuzu) that can answer questions about my work. It runs on Gemini 2.5 Flash via OpenRouter, with a custom RAG pipeline on top of Convex — projects, blog posts, and work history are all embedded into a single namespace and retrieved per-query through targeted tools (`listProjects`, `getProjectDetails`, `searchPortfolio`).

## The architecture

Content is the interesting part. Project data lives in MDX files inside the repo — that's the **single source of truth**. On every Vercel deploy, a `postbuild` script reads those MDX files, formats them, and pushes structured payloads to a Convex action that re-ingests them into both the documents table (for tool responses) and the RAG vector store (for semantic search). Media (videos, gifs, screenshots) lives on ImageKit; the MDX frontmatter just holds the URLs.

The result: editing a project means editing one MDX file. The deploy handles everything else.
